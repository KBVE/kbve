# `apps/kube/agones/factorio` — Phase 1 GameServer deployment

ArgoCD-managed Agones `GameServer` for the KBVE Factorio dedicated server.
Single-pod, two containers (factorio + relay), `Static` port policy on
UDP 34197, RCON loopback-only, saves on a `longhorn` PVC.

Tracks issue [#11138](https://github.com/KBVE/kbve/issues/11138). Phase
ordering and what's deferred is documented at the bottom.

## What ships on a clean apply

| Manifest                                 | Purpose                                                         |
| ---------------------------------------- | --------------------------------------------------------------- |
| `manifests/namespace.yaml`               | `factorio` namespace                                            |
| `manifests/saves-pvc.yaml`               | `factorio-saves` Longhorn PVC, 10 Gi, RWO                       |
| `manifests/gameserver.yaml`              | Agones `GameServer`, two containers, mounts PVC + emptyDir      |
| `manifests/service.yaml`                 | UDP `LoadBalancer` on 34197, selects on `agones.dev/gameserver` |
| `manifests/rcon-sealed-secret.yaml`      | _generated by `seal-rcon.sh`, see below_                        |
| `manifests/irc-creds-sealed-secret.yaml` | _generated by `seal-irc-creds.sh`, see below_                   |
| `application.yaml`                       | ArgoCD `Application` pointing at `manifests/`                   |

## First-deploy runbook

1. **Seal the matchmaking token** (local laptop, with `kubectl` pointed
   at the prod cluster and `kubeseal` installed):

    ```bash
    ./seal-credentials.sh  # prompts for matchmaking username + token (+ optional game password)
    ```

    Writes `manifests/credentials-sealed-secret.yaml`. Commit it — the
    sealed payload is safe in git. The token is pulled from
    <https://www.factorio.com/profile>; rotate it there if it ever
    leaks (a fresh token instantly invalidates the previous one).

    **RCON password is generated, not sealed.**
    `manifests/rcon-generated-secret.yaml` declares a
    `generators.external-secrets.io/v1alpha1/Password` + an
    `ExternalSecret` with `refreshInterval: 0`. On first reconcile the
    ExternalSecrets operator generates a random 48-char password into
    the `factorio-rcon` Secret. Nobody types it; nobody rotates it on a
    schedule. To rotate manually: delete the Secret and the
    GameServer pod — operator regenerates, Agones respawns the pod, the
    relay re-auths against the new password on next boot.

    **IRC creds are not needed.**
    The relay sidecar talks to the internal kbve IRC server
    (`ergo-irc-service.irc.svc.cluster.local:6667`) inside the
    cluster. The connection is anonymous (no NickServ identify), so
    `gameserver.yaml` sets `IRC_SERVER` / `IRC_PORT` / `IRC_USE_TLS` /
    `IRC_NICK` / `IRC_CHANNEL` as static env values — no SealedSecret,
    no kubeseal step.

2. **Apply the ArgoCD `Application`**:

    The Application is already registered in the kustomize app-of-apps
    (`apps/kube/kustomization.yaml`), so `kube-root` creates it on the next
    reconcile — no manual `kubectl apply` is needed once that PR is in.

    ArgoCD picks up everything under `manifests/`. The sealed-secrets
    controller decrypts the sealed payloads into regular `Secret`s named
    `factorio-rcon`, `factorio-irc`, and `factorio-credentials` inside the
    `factorio` namespace, and the GameServer pod mounts them.

3. **Verify**:

    ```bash
    kubectl -n factorio get gameserver
    kubectl -n factorio get pods
    kubectl -n factorio logs -l app=factorio -c factorio --tail=100
    kubectl -n factorio logs -l app=factorio -c factorio-relay --tail=100
    ```

    GameServer state should reach `Ready` once `agones-shim` posts
    `Ready()` ~45 s after factorio binds UDP 34197.

## Loading a save (Robots.zip)

By default the GameServer boots into the baked-in `kbve` scenario. To load
a pre-existing save instead:

1. Once the GameServer is running, copy the save onto the PVC via the
   factorio container:

    ```bash
    POD=$(kubectl -n factorio get pods -l app=factorio -o name | head -1)
    kubectl -n factorio cp Robots.zip "${POD#pod/}:/factorio/saves/Robots.zip" -c factorio
    ```

2. Edit `manifests/gameserver.yaml` and uncomment the `FACTORIO_SAVE` env
   block:

    ```yaml
    - name: FACTORIO_SAVE
      value: 'Robots.zip'
    ```

3. Commit + push. ArgoCD restarts the GameServer; the shim sees
   `FACTORIO_SAVE` and switches the launch flag from
   `--start-server-load-scenario kbve` to `--start-server /factorio/saves/Robots.zip`.

Saves stay on the PVC across pod respawns; only deleting the PVC drops
them. The PVC is RWO, so only one GameServer pod can hold the save at a
time — fine for the single-pod Phase 1 layout.

## Operational notes

- **Why `Static` port policy** — players connect to a known
  `factorio.kbve.com:34197`, no Allocator API or matchmaking lookup needed.
  When we move to a `Fleet` of >1 (PvP shard, vanilla shard, etc.) this
  becomes `Dynamic` and players will hit a lobby endpoint that proxies the
  allocation; that's out of scope for Phase 1.
- **Why loopback-only RCON** — RCON has no rate limiting and the password
  grants admin / arbitrary Lua execution. The relay sidecar shares the pod
  netns, so loopback is the only surface that needs to expose it. Do not
  add a Service for `27015/tcp`.
- **Why an in-memory `emptyDir` for `/shared/log`** — the relay tails this
  for the chat bridge; the volume is short-lived (pod lifetime), small
  (capped at 64 Mi), and ramfs avoids disk write amplification.
- **Why `externalTrafficPolicy: Local`** — preserves the client source IP
  in the kernel UDP socket so admin tooling that bans / rate-limits by IP
  doesn't see every connection as coming from the same node.
